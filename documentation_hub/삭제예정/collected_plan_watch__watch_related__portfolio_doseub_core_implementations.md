# 두섭이 포트폴리오 핵심 구현 4가지 추출본

## 01 Watch-iPhone 양방향 동기화 충돌 해결

### 타임스탬프 기반 병합 전략으로 데이터 손실 없는 양방향 동기화 구현

### 문제
Watch와 iPhone이 오프라인 상태에서 각각 동일한 서핑 기록을 수정한 뒤 재연결 시 수정본이 엇갈리며 기록이 중복되거나 유실되었다. 단순 병합을 시도하면 동일 레코드가 두 건으로 분기됐다. 특히 한쪽에서 기록을 삭제하고 다른 쪽에서 수정한 경우, 삭제 정보가 동기화되지 않아 이미 삭제된 기록이 다시 나타나는 현상도 발생했다. WatchConnectivity의 비동기 특성상 두 디바이스가 동기화 시점을 예측하기 어려워 충돌 해결 전략 없이는 데이터 무결성을 보장할 수 없었다.

### 접근
타임스탬프를 기준으로 `최신 우선 병합 + 삭제 전파` 전략을 채택했다.
각 기록에 `lastModifiedAt`과 `deviceID`를 추가해 수정 출처를 추적하고, 삭제는 `isDeleted` 플래그로 처리해 삭제 정보가 동기화되도록 했다.  
전체 데이터가 아닌 변경된 필드만 전송하는 증분 업데이트를 적용했다.  
충돌 해결 로직은 `삭제 우선 → 최신 수정 시각 우선 → 디바이스 식별자 사전순` 규칙을 적용했다. 여러 기록은 배치로 묶어 전송했고 실패 시 재시도 로직을 추가했다.

### 결과
동시 수정이 발생해도 데이터 손실/중복 없이 일관된 기록을 유지했고, 동기화 충돌로 인한 크래시는 0건으로 감소했다.  
증분 업데이트로 메시지 크기가 평균 70% 감소해 배터리 소모가 줄고, 배치 동기화로 전송 횟수가 60% 감소해 블루투스 부하가 낮아졌다.

## 02 지역별 차트 조회 성능 개선

### LRU 메모리 캐시와 예측 prefetch로 조회 속도 단축 및 비용 절감

### 문제
여러 지역을 연속으로 조회할 때 지역마다 Firestore 요청이 반복되어 대기 시간이 누적되었다.  
5개 지역 비교 시 총 대기 시간이 약 6초까지 늘어 사용자 흐름이 끊겼고, 계절성 사용량이 늘면 읽기 요청이 급증해 비용 리스크도 커졌다.

### 접근
- `NSCache` 기반 LRU 캐시로 최근 조회 지역 데이터 재사용
- 세션 패턴을 활용한 인접 지역 예측 prefetch 적용
- 캐시 데이터는 즉시 표시하고 백그라운드에서 최신 데이터 갱신하는 stale-while-revalidate 적용
- 캐시 로직을 `ChartCacheManager`로 캡슐화하여 화면 코드와 분리
- 세션 패턴 기준 캐시 상한값 설정으로 메모리 사용 통제

### 결과
- 캐시 히트율 70% 달성으로 Firestore 읽기 요청 50% 감소
- 차트 표시 시간: 1.2초 → 0.3초
- prefetch 적용 후 다음 지역 즉시 표시 비율 60% 도달
- 불필요한 읽기 감소로 월 비용 35% 절감

### Implementation Notes
- NSCache LRU, 최대 10개 항목 캐싱
- 다음 조회 후보 3개 지역 prefetch
- 최신 데이터는 해시 비교 후 변경 시에만 UI 업데이트

## 03 Live Activities와 Dynamic Island 기반 실시간 세션 모니터링

### 잠금 화면 통합으로 앱 실행 없는 세션 상태 확인 구현

### 문제
서핑 세션 중에는 iPhone을 방수 케이스에 두고 바다로 들어가는 사용자가 많아, 세션 상태 확인을 위해 앱을 열어야 하는 마찰이 컸다. 상태 확인에 필요한 동작이 즉시성의 핵심이어서, 앱 실행 여부가 정보 접근을 방해했다.

### 접근
iOS 16 `Live Activities`와 `Dynamic Island`(ActivityKit)를 적용해 잠금 화면/노치 영역에서 실시간 세션 정보를 노출했다.  
`ActivityAttributes`로 고정 정보(시작 시간, 해변 이름)와 동적 정보(경과 시간, 라이딩 횟수, 평균 심박수) 를 분리해 업데이트 효율을 개선했다.
- 경과 시간: 1분 간격
- 라이딩 횟수: 즉시
- 심박수: 10초 간격

Widget Extension의 SwiftUI로 Dynamic Island 축소/확장/최소 상태 레이아웃을 구성하고, 라이딩 횟수 증가 시 페이드 애니메이션, 심박수 임계값 초과 시 펄스 애니메이션 적용했다.

### 결과
앱 오픈 없이도 잠금 화면에서 세션 상태 확인이 가능해져 사용자 참여도 상승, 방수 케이스 개봉 빈도 감소로 물 손상 위험 완화, 실시간 피드백으로 동기부여 효과 개선.  
배터리 최적화를 위해  
- 세션 초반 1분 간격
- 30분 후 5분 간격
- 1시간 후 10분 간격  
으로 갱신 주기 조정해 배터리 수명 약 25% 연장.

## 04 목업 데이터 기반 개발 전략

### 외부 API 중단 상황에서 개발 연속성 확보

### 문제
기상청 공공데이터포털 API 중단(2025.09.26~)으로 실시간 파도/바람/기상 기반 차트 렌더링, 날씨 분류, 저장 기능 개발이 멈추어 2주 이상 일정 지연이 예상됐다.

### 접근
- API 중단 전 응답 구조를 분석해 Firestore에 목업 데이터셋 구축
- 프로토콜 기반 추상화로 비즈니스 로직 계층이 데이터 소스를 인지하지 않게 설계
- 데이터 접근 인터페이스를 프로토콜화, ViewModel/UseCase는 실제 API 여부를 알지 못하게 분리
- DTO를 기상청 API 필드 전부 Optional 처리해 응답 가변성 대응
- RxSwift 기반 비동기 구조 유지, 목업 저장은 4개 지역 11개 해변 72시간 예보(3시간 간격) 반영

### 결과
API 중단 기간에도 핵심 기능을 멈추지 않고 개발을 지속해 일정 지연을 회피했고, 복구 후 실제 데이터 수집 로직만 재개하면 됐다.  
계층 분리와 의존성 주입 설계 덕분에 코드 수정 없이 데이터 소스 전환이 가능해졌으며, 외부 장애 상황 대비 역량을 확보했다.
