## 핵심 구현


## 01 Watch-iPhone 양방향 동기화 충돌 해결
타임스탬프 기반 병합 전략으로 데이터 손실 없는 양방향 동기화 구현

**문제**
Watch와 iPhone이 오프라인 상태에서 각각 동일한 서핑 기록을 수정한 뒤 재연결 시 수정본이 엇갈리며 기록이 중복되거나 유실되었다. 단순 병합을 시도하면 동일 레코드가 두 건으로 분기됐다. 특히 한쪽에서 기록을 삭제하고 다른 쪽에서 수정한 경우, 삭제 정보가 동기화되지 않아 이미 삭제된 기록이 다시 나타나는 현상도 발생했다. WatchConnectivity의 비동기 특성상 두 디바이스가 `동기화 시점을 예측하기 어려워`, 충돌 해결 전략 없이는 `데이터 무결성을 보장할 수 없었다`.

**접근**
타임스탬프를 기준으로 `최신 우선 병합 + 삭제 전파` 전략을 채택했다.  `각 기록에 lastModifiedAt과 deviceID를 추가해 수정 출처를 추적하고, 삭제는 isDeleted 플래그로 처리해 삭제 정보가 동기화되도록 했다.` 이 방식을 선택한 이유는 두 기기 모두 기록을 수정할 수 있어 단일 진실 소스를 고정할 수 없었고, 충돌 규칙이 명확해야 유지보수와 재현이 쉬웠기 때문이다. 
 전체 데이터가 아닌 변경된 필드만 전송하는 `증분 업데이트`를 적용하였다. 충돌 해결 로직은  삭제 우선 → 최신 수정 시각 우선 → 디바이스 식별자 사전순 선택 규칙을 적용했다. 여러 기록은 배치로 묶어 전송했고, 실패 시 재시도 로직을 추가했다.

**결과**
Watch와 iPhone에서 동시 수정이 발생해도 데이터 손실이나 중복 없이 일관된 기록이 유지되었고, 동기화 충돌로 인한 크래시가 0건으로 감소했다. 증분 업데이트로 메시지 크기가 평균 70% 감소하여 배터리 소모가 줄었고, 배치 동기화는 메시지 전송 횟수를 60% 감소시켜 블루투스 부하를 낮췄다. 


## 02 지역별 차트 조회 성능 개선
LRU 메모리 캐시와 예측 prefetch로 조회 속도 단축 및 비용 절감

**문제**
서퍼들이 당일 최적의 스팟을 찾기 위해 여러 지역을 연속으로 조회할 때, 지역마다 Firestore 요청이 반복되어 대기 시간이 누적됐다.  
5개 지역을 비교하면 총 대기 시간이 약 6초까지 늘어 사용 흐름이 끊겼다.  
이미 조회한 지역을 다시 선택해도 동일 데이터를 재요청하는 구조였다.  
시즌 사용량을 가정하면 읽기 요청이 급증해 비용 리스크가 있었다.

**접근**
NSCache 기반 LRU 캐시로 최근 조회한 지역의 차트 데이터를 재사용했다.  
다음 선택 가능성이 높은 인접 지역을 예측해 prefetch로 미리 로드했다.  
캐시 데이터는 즉시 표시하고 백그라운드에서 최신 데이터를 갱신하는 stale-while-revalidate 패턴을 적용했다.  
캐시 로직은 ChartCacheManager로 캡슐화해 화면 코드와 분리했다.  
세션 패턴 기준으로 캐시 상한을 두어 메모리 사용을 통제했다.

**결과**
캐시 히트율 70%를 달성해 Firestore 읽기 요청이 50% 감소했다.  
차트 표시 시간이 1.2초에서 0.3초로 단축됐다.  
prefetch 적용 후 다음 지역 선택 시 즉시 표시 비율이 60%까지 올라갔다.  
불필요한 읽기가 줄어 월 비용이 35% 절감됐다.

**Implementation Notes**
- NSCache LRU, 최대 10개 항목 캐싱
- 다음 조회 후보 3개 지역 prefetch
- 최신 데이터는 해시 비교 후 변경 시에만 UI 업데이트


## 03 Live Activities와 Dynamic Island 기반 실시간 세션 모니터링

잠금 화면 통합으로 앱 실행 없는 세션 상태 확인 구현

**문제**
서핑 세션 진행 중 사용자는 iPhone을 방수 케이스에 넣어 해변에 두고 바다로 들어가므로, 세션 진행 상황을 확인하려면 젖은 손으로 iPhone을 꺼내 잠금 해제 후 앱을 열어야 했다. 전통적으로 앱은 사용자가 명시적으로 열어야만 정보를 볼 수 있어, 실시간으로 진행되는 활동을 추적하는 앱에게는 치명적인 제약이었다. 사용자는 "지금 몇 분째 서핑 중인지", "몇 번의 파도를 탔는지" 같은 정보를 빠르게 확인하고 싶어 했지만, 앱을 열기까지의 마찰이 너무 컸고, 서핑처럼 시간에 민감한 활동에서는 지속적인 상태 확인을 위해 이 마찰을 줄이는 것이 핵심 사용자 경험 문제였다.

**접근**
iOS 16의 Live Activities와 Dynamic Island를 ActivityKit 프레임워크로 구현하여 잠금 화면과 노치 영역에서 실시간 세션 정보를 표시했다. ActivityAttributes 프로토콜로 고정 정보(시작 시간, 해변 이름)와 동적 정보(경과 시간, 라이딩 횟수, 평균 심박수)를 분리하여 업데이트 효율성을 높였고, 경과 시간은 1분 간격, 라이딩 횟수는 즉시, 심박수는 10초 간격으로 업데이트하여 정보 신선도와 배터리 효율의 균형을 맞췄다. Widget Extension에서 SwiftUI로 Dynamic Island의 축소/확장/최소 상태별 레이아웃을 구성하고, 라이딩 횟수 증가 시 페이드 애니메이션과 심박수 임계값 초과 시 펄스 애니메이션을 적용하여 변화를 즉시 인지하게 했다. 배터리 최적화를 위해 세션 초반 1분 간격, 30분 후 5분 간격, 1시간 후 10분 간격으로 업데이트 빈도를 동적 조정했고, Live Activity 탭 시 딥링크로 세션 상세 화면 이동, 길게 누르면 조기 종료 옵션을 제공했다.

**결과**
사용자가 세션 상태를 확인하는 빈도가 약 5배 증가했고, 앱을 열지 않고도 잠금 화면에서 즉시 확인 가능해져 사용자 참여도가 높아졌다. iPhone을 방수 케이스에서 꺼내는 빈도가 감소하여 물 손상 위험이 낮아졌고, 라이딩 횟수 실시간 확인으로 "한 번 더 타자"는 동기 부여 효과가 발생했다. 업데이트 빈도 동적 조정으로 배터리 수명이 약 25% 연장되었으며, Dynamic Island 지원으로 앱이 최신 iOS 기능을 적극 활용한다는 긍정적 이미지를 형성했다. 사용자는 앱을 열지 않고도 필요한 정보를 즉시 확인할 수 있게 되어, 기술이 흐름을 방해하지 않고 자연스럽게 통합되었다.

## 04 목업 데이터 기반 개발 전략

외부 API 중단 상황에서 개발 연속성 확보

**문제**
2025년 9월 26일 국가정보자원관리원 화재로 기상청 공공데이터포털 API가 약 2주간 중단되었다. DoSurf는 실시간 파도, 바람, 기상 데이터에 전적으로 의존하는 서비스로, API 중단 시 신규 기능 개발과 테스트가 불가능해졌다. 차트 렌더링, 날씨 분류 로직, 데이터 저장 등 핵심 기능 개발이 모두 멈춰 2주 이상의 개발 일정 지연이 불가피한 상황이었다.

**접근**
API 중단 이전 수집한 실제 응답 구조를 분석하여 Firestore에 목업 데이터 세트를 구축했다. 프로토콜 기반 추상화 설계로 앱의 비즈니스 로직 계층이 데이터 출처를 인지하지 못하도록 격리했다. 데이터 접근 인터페이스를 프로토콜로 정의하고 실제 구현체가 이를 준수하도록 하여, ViewModel과 UseCase가 실제 API인지 목업 데이터인지 알 수 없게 만들었다. DTO는 기상청 API의 모든 필드를 Optional로 정의하여 실제 응답의 가변성을 반영했고, RxSwift 기반 비동기 처리 구조를 그대로 유지했다. 목업 데이터는 강릉, 포항, 제주, 부산 4개 지역 11개 해변의 3시간 간격 72시간 예보를 Firestore에 저장하여 실제 프로덕션 환경과 동일한 쿼리 방식을 보장했다.

**결과**
API 중단 기간 동안 개발을 멈추지 않고 차트 렌더링, 날씨 타입 분류, 서핑 기록 저장 등 핵심 기능을 완성했다. ## 02 공용 데이터 계층 도입으로 앱과 위젯 상태 동기화

앱/위젯이 다른 저장소를 쓰는 구조를 하나로 단일화

- **문제**

  Core Data 스키마가 변경될 때마다 앱과 위젯 저장소를 각각 마이그레이션해야 했고, 이 과정에서 데이터 이관 누락이나 기록 유실 위험도 존재했다. 사용자 입장에서는 어느 화면의 정보가 정확한지 혼란을 겪을 수밖에 없는 구조였다.

- **접근**  
  앱과 위젯이 동일한 복용 상태를 사용하도록 공용 데이터 계층을 도입했다. App Group을 활용해 하나의 Core Data 저장소를 앱과 위젯이 함께 참조하도록 구성했다.

- **결과**

## 03 Diffable Data Source 기반 단일 셀 갱신 구조

캘린더 상태 변경 시 필요한 셀만 안정적으로 업데이트하는 UI 구조 개선

- **문제**

  또한 IndexPath를 기준으로 셀을 직접 참조하는 구조였기 때문에, 데이터 변경 타이밍이 어긋나면 UI 업데이트 중 크래시가 발생할 위험도 있었다. 사용자 경험과 안정성 모두에서 개선이 필요한 상태였다.

- **접근**  
  날짜 단위를 하나의 독립된 아이템으로 모델링해, 특정 날짜의 상태만 개별적으로 갱신할 수 있는 구조로 변경했다.  
  컬렉션 뷰는 Diffable Data Source를 사용해 데이터 변경 사항을 안전하게 계산하도록 했고, 전체 화면을 다시 그리는 방식 대신 변경된 항목만 반영되도록 구성했다.  
  상태 변경 시에는 해당 날짜 셀만 다시 설정되도록 처리해 불필요한 UI 갱신을 줄였다. 복용 상태 데이터는 ViewModel에서 단일 흐름으로 관리하고, UI는 이 변경 결과만 반영하도록 역할을 분리했다.  
  이 구조를 통해 화면 깜빡임과 업데이트 충돌 가능성을 동시에 해소했다.

- **결과**

  Diffable Data Source가 항목의 식별자를 기준으로 변경 사항을 계산하므로, IndexPath 불일치로 인한 크래시 위험도 제거되었다.  
  데이터 변경과 UI 갱신이 단방향 흐름으로 정리되어, 다른 화면이나 기능에서 상태를 변경해도 동일한 업데이트 경로를 재사용할 수 있게 되었다.  
  결과적으로 캘린더 화면의 안정성과 사용자 체감 품질이 함께 개선되었다.

## 04 의료 규칙 기반 단위 테스트

  또한 우선순위 기반 메시지 규칙이 충돌하지 않는지 확인하는 테스트를 추가해, 규칙 간 상호작용까지 함께 검증했다.

# Page 10 / 10 — 키득 (KIDK) 핵심 구현 (Compact)

## 01 REST API 통신 계층과 토큰 인터셉터 기반 인증 자동화
- 문제: Spring + AWS 기반 REST API 20개 엔드포인트의 응답 형식이 제각각이라 디코딩과 에러 처리가 화면마다 달랐다. 동일한 401도 만료와 권한 부족을 구분해야 했고, Firebase 로그인 뒤 백엔드 JWT를 교환해 Access/Refresh 토큰을 요청마다 맞게 첨부해야 했다. 초기 구현은 Codable 정의와 분기 처리로 API 1개당 평균 2시간이 소요되었고, 20개로 확장 시 일정 리스크가 컸다.
- 접근: 백엔드와 협의해 응답을 ApiResponse<T>로 통일하고 NetworkService에서 공통 디코딩을 수행했다. 성공은 data를 반환하고 실패는 HTTP 상태 코드와 error.code를 함께 매핑해 NetworkError로 변환했다. 이 방식을 선택한 이유는 ViewModel에서 처리해야 할 분기와 중복을 줄여야 했고, API 수가 많아질수록 표준화의 효율이 커졌기 때문이다. 인증은 RequestInterceptor 패턴의 adapt 단계에서 자동으로 헤더를 추가하도록 구성했다. requiresAuth/usesRefreshToken 플래그로 Access 또는 Refresh 토큰을 구분해 실수 가능성을 낮췄고, TokenManager가 Keychain에서 최신 값을 읽도록 했다.
- 결과: 디코딩 실패율은 15%에서 0.5% 미만으로 감소했고, 에러 원인 파악 시간은 평균 30분에서 5분으로 단축되었다. API 1개당 개발 시간이 2시간에서 20분 수준으로 줄어 20개 엔드포인트를 1주일 내 완성했다. 모든 화면이 동일한 에러 처리 로직을 공유해 유지보수가 쉬워졌고, 토큰 타입 실수와 인증 관련 버그가 줄었다.

## 02 Keychain 기반 토큰 보안 저장
- 문제: 초기에는 Access Token과 Refresh Token을 UserDefaults에 저장했다. 자동 로그인 기능을 테스트하는 동안에는 문제가 없어 보였고, 앱을 종료하고 다시 열어도 토큰이 유지되어 사용자가 로그인 화면을 거치지 않고 메인 화면으로 진입했다. 문제는 보안 검토 과정에서 드러났다. UserDefaults는 평문으로 저장되어 탈옥된 기기나 백업 파일에서 데이터를 읽을 수 있었고, Access Token이 노출되면 해당 사용자로 위장한 API 호출이, Refresh Token까지 노출되면 무한정 토큰 갱신이 가능했다. KIDK는 아동의 용돈 데이터를 다루므로 토큰 탈취는 개인정보 유출로 직결되었다.
- 접근: 시스템 레벨 암호화 저장소인 Keychain을 도입하여 토큰을 보관했다. 보안 API를 래핑한 클래스를 만들고 Access Token과 Refresh Token을 구분하여 저장했으며, 저장 시도 시 이미 존재하는 항목은 자동으로 업데이트하도록 했다. 저장소 인터페이스를 분리하여 테스트 환경에서는 실제 보안 저장소 대신 임시 메모리를 사용하도록 했다. 로그아웃 기능은 두 토큰을 모두 삭제하는 단일 메서드로 구현했으며, 서버 요청 실패 여부와 관계없이 로컬 데이터는 항상 정리되도록 보장했다.
- 결과: Keychain 도입으로 토큰 탈취 위험이 제거되었다. UserDefaults 사용 시 백업 파일에서 평문 토큰을 추출할 수 있었으나, Keychain 적용 후 암호화된 저장소로 보호되어 동일한 공격 방법이 차단되었다. 탈옥 기기에서도 추가 공격 없이는 토큰에 접근할 수 없게 되었고, 아동 용돈 데이터를 다루는 앱의 보안 요구사항을 충족했다. 저장소 추상화를 통해 다른 민감 데이터도 동일한 보안 수준으로 보호할 수 있는 기반이 마련되었다.

## 03 Firebase-백엔드 이중 인증 흐름 통합
- 문제: Firebase Authentication과 백엔드 JWT를 함께 쓰는 구조라 사용자는 Firebase로 로그인하지만, 실제 API 호출에는 백엔드가 발급한 Access/Refresh 토큰이 필요했다. Firebase 로그인만으로는 API를 호출할 수 없고, 백엔드 로그인만 구현하면 Firebase 보안 이점을 잃는다. ID Token과 UID를 받은 뒤 백엔드에서 JWT를 교환해 저장하는 순차 흐름이 필요했으며, 단계별 실패 원인을 구분해 안내해야 했다.
- 접근: Firebase 비동기 처리를 RxSwift 스트림으로 변환해 로그인 성공 시 백엔드 로그인으로 이어지는 2단계 체인을 구성했다. 이 방식을 선택한 이유는 중첩 콜백을 줄이고, 실패 지점을 명확히 구분해 사용자 메시지를 일관되게 처리하기 위해서다. 성공 시 사용자 정보와 토큰을 저장하고, 실패는 동일한 에러 경로로 전파해 화면에서 분기 처리를 최소화했다. 자동 로그인은 저장된 설정을 불러오고, 로그인 성공 시 이메일을 저장해 다음 실행 시 자동 채움이 되도록 했다.
- 결과: Firebase 로그인 후 백엔드 로그인을 순차적으로 실행하는 흐름이 명확해졌고, 실패 지점을 구분해 안내할 수 있었다. 로그인 UI는 단일 경로로 정리되어 유지보수가 쉬워졌으며, 이메일 자동 채움으로 사용자가 비밀번호만 입력하면 되는 경험을 제공했다. 두 인증 시스템이 한 절차로 연결되어 사용자는 별도의 추가 인증을 인지하지 않게 되었다.

## 04 Vision 프레임워크 기반 영수증 OCR 자동 파싱
- 문제: 영수증 금액과 상호명을 수동 입력해야 해 초등학생 사용자의 이탈이 잦았고, 숫자 입력 실수가 잔액 왜곡으로 이어졌다. OCR을 도입하면 입력 시간을 줄일 수 있으나, 영수증 구김이나 반사로 인식 정확도가 떨어져 잘못된 금액이 저장될 위험이 있었다. 자동화와 사용자 확인의 균형이 필요했다.
- 접근: Vision 텍스트 인식을 accurate 모드로 설정하고 Document Camera로 영수증 경계를 자동 보정했다. 처리 시간을 조금 늘리더라도 금액 인식의 정확도가 중요하다고 판단했기 때문이다. 결제 금액은 “합계/총액/결제금액/total” 키워드 주변의 숫자를 우선 추출하고, 키워드가 없으면 가장 큰 숫자를 선택했다. 상호명은 첫 5줄에서 유효 길이의 가장 긴 줄을 선택해 전화번호나 주소 오인식을 줄였다. 결과는 입력 필드에 자동 채우되 사용자가 수정할 수 있도록 하고, 모든 필드가 채워져야 저장이 가능하도록 검증했다.
- 결과: 평균 입력 시간이 30초에서 8초로 75% 단축되고 오류율이 5%에서 1%로 감소했다. 텍스트 인식률은 75%에서 95%로 개선되었고, 다양한 영수증 포맷에서도 안정적으로 금액과 상호명을 추출했다. 자동 입력 후 확인 흐름으로 잘못된 금액 저장을 줄여 잔액 정확성을 유지했으며, 거래 기록 지속률이 높아졌다.
