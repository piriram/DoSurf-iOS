## 스크린샷 라벨

- 앱 사용 가이드 화면

## 기획 의도
- Problem: `초보 서퍼`는 복잡한 차트 때문에 오늘의 서핑 가능 여부를 빠르게 결정하기 어렵다.
- Solution: 해양 차트를 `직관적인 UI`로 보여주고, 내 기록 기반으로 `나에게 맞는 서핑 조건`을 보여준다.


## 프로젝트 정보
- 개발 기간: 2025.04 ~ 2025.6 (3개월)
- 플랫폼: iOS 16.0+
- 역할: `1인 iOS 개발 및 백엔드(Firebase) 구축`
- 상태: App Store 배포
- 태그: 해양 차트, 서핑 기록, 개인화 통계

## 핵심 기능
1. 실시간 해양 차트: 오늘 파도 차트를 한눈에 확인
2. 서핑 기록 및 통계: 내가 좋았던 조건을 기록으로 저장하고 통계로 확인
3. Apple Watch 연동: 심박수와 이동 경로를 수집해 세션 기록으로 저장
4. Live Activities: 잠금 화면과 Dynamic Island에서 세션 상태를 실시간 표시


## 기술 스택
- 아키텍쳐: MVVM-Coordinator, Clean Architecture
- UI: UIKit, SnapKit, ActivityKit
- 데이터: Core Data, HealthKit
- 디바이스 연동: WatchConnectivity, ActivityKit

## 핵심 구현


## 01 Watch-iPhone 양방향 동기화 충돌 해결
타임스탬프 기반 병합 전략으로 데이터 손실 없는 양방향 동기화 구현

**문제**
Watch와 iPhone이 오프라인 상태에서 각각 동일한 서핑 기록을 수정한 뒤 재연결 시 수정본이 엇갈리며 기록이 중복되거나 유실되었다. 단순 병합을 시도하면 동일 레코드가 두 건으로 분기됐다. 특히 한쪽에서 기록을 삭제하고 다른 쪽에서 수정한 경우, 삭제 정보가 동기화되지 않아 이미 삭제된 기록이 다시 나타나는 현상도 발생했다. WatchConnectivity의 비동기 특성상 두 디바이스가 `동기화 시점을 예측하기 어려워`, 충돌 해결 전략 없이는 `데이터 무결성을 보장할 수 없었다`.

**접근**
타임스탬프를 기준으로 `최신 우선 병합 + 삭제 전파` 전략을 채택했다.  `각 기록에 lastModifiedAt과 deviceID를 추가해 수정 출처를 추적하고, 삭제는 isDeleted 플래그로 처리해 삭제 정보가 동기화되도록 했다.` 이 방식을 선택한 이유는 두 기기 모두 기록을 수정할 수 있어 단일 진실 소스를 고정할 수 없었고, 충돌 규칙이 명확해야 유지보수와 재현이 쉬웠기 때문이다. 
 전체 데이터가 아닌 변경된 필드만 전송하는 `증분 업데이트`를 적용하였다. 충돌 해결 로직은  삭제 우선 → 최신 수정 시각 우선 → 디바이스 식별자 사전순 선택 규칙을 적용했다. 여러 기록은 배치로 묶어 전송했고, 실패 시 재시도 로직을 추가했다.

**결과**
Watch와 iPhone에서 동시 수정이 발생해도 데이터 손실이나 중복 없이 일관된 기록이 유지되었고, 동기화 충돌로 인한 크래시가 0건으로 감소했다. 증분 업데이트로 메시지 크기가 평균 70% 감소하여 배터리 소모가 줄었고, 배치 동기화는 메시지 전송 횟수를 60% 감소시켜 블루투스 부하를 낮췄다. 


## 02 지역별 차트 조회 성능 개선
LRU 메모리 캐시와 예측 prefetch로 조회 속도 단축 및 비용 절감

**문제**
서퍼들이 당일 최적의 스팟을 찾기 위해 여러 지역을 연속으로 조회할 때, 지역마다 Firestore 요청이 반복되어 대기 시간이 누적됐다.  
5개 지역을 비교하면 총 대기 시간이 약 6초까지 늘어 사용 흐름이 끊겼다.  
이미 조회한 지역을 다시 선택해도 동일 데이터를 재요청하는 구조였다.  
시즌 사용량을 가정하면 읽기 요청이 급증해 비용 리스크가 있었다.

**접근**
NSCache 기반 LRU 캐시로 최근 조회한 지역의 차트 데이터를 재사용했다.  
다음 선택 가능성이 높은 인접 지역을 예측해 prefetch로 미리 로드했다.  
캐시 데이터는 즉시 표시하고 백그라운드에서 최신 데이터를 갱신하는 stale-while-revalidate 패턴을 적용했다.  
캐시 로직은 ChartCacheManager로 캡슐화해 화면 코드와 분리했다.  
세션 패턴 기준으로 캐시 상한을 두어 메모리 사용을 통제했다.

**결과**
캐시 히트율 70%를 달성해 Firestore 읽기 요청이 50% 감소했다.  
차트 표시 시간이 1.2초에서 0.3초로 단축됐다.  
prefetch 적용 후 다음 지역 선택 시 즉시 표시 비율이 60%까지 올라갔다.  
불필요한 읽기가 줄어 월 비용이 35% 절감됐다.

**Implementation Notes**
- NSCache LRU, 최대 10개 항목 캐싱
- 다음 조회 후보 3개 지역 prefetch
- 최신 데이터는 해시 비교 후 변경 시에만 UI 업데이트


## 03 Live Activities와 Dynamic Island 기반 실시간 세션 모니터링

잠금 화면 통합으로 앱 실행 없는 세션 상태 확인 구현

**문제**
서핑 세션 진행 중 사용자는 iPhone을 방수 케이스에 넣어 해변에 두고 바다로 들어가므로, 세션 진행 상황을 확인하려면 젖은 손으로 iPhone을 꺼내 잠금 해제 후 앱을 열어야 했다. 전통적으로 앱은 사용자가 명시적으로 열어야만 정보를 볼 수 있어, 실시간으로 진행되는 활동을 추적하는 앱에게는 치명적인 제약이었다. 사용자는 "지금 몇 분째 서핑 중인지", "몇 번의 파도를 탔는지" 같은 정보를 빠르게 확인하고 싶어 했지만, 앱을 열기까지의 마찰이 너무 컸고, 서핑처럼 시간에 민감한 활동에서는 지속적인 상태 확인을 위해 이 마찰을 줄이는 것이 핵심 사용자 경험 문제였다.

**접근**
iOS 16의 Live Activities와 Dynamic Island를 ActivityKit 프레임워크로 구현하여 잠금 화면과 노치 영역에서 실시간 세션 정보를 표시했다. ActivityAttributes 프로토콜로 고정 정보(시작 시간, 해변 이름)와 동적 정보(경과 시간, 라이딩 횟수, 평균 심박수)를 분리하여 업데이트 효율성을 높였고, 경과 시간은 1분 간격, 라이딩 횟수는 즉시, 심박수는 10초 간격으로 업데이트하여 정보 신선도와 배터리 효율의 균형을 맞췄다. Widget Extension에서 SwiftUI로 Dynamic Island의 축소/확장/최소 상태별 레이아웃을 구성하고, 라이딩 횟수 증가 시 페이드 애니메이션과 심박수 임계값 초과 시 펄스 애니메이션을 적용하여 변화를 즉시 인지하게 했다. 배터리 최적화를 위해 세션 초반 1분 간격, 30분 후 5분 간격, 1시간 후 10분 간격으로 업데이트 빈도를 동적 조정했고, Live Activity 탭 시 딥링크로 세션 상세 화면 이동, 길게 누르면 조기 종료 옵션을 제공했다.

**결과**
사용자가 세션 상태를 확인하는 빈도가 약 5배 증가했고, 앱을 열지 않고도 잠금 화면에서 즉시 확인 가능해져 사용자 참여도가 높아졌다. iPhone을 방수 케이스에서 꺼내는 빈도가 감소하여 물 손상 위험이 낮아졌고, 라이딩 횟수 실시간 확인으로 "한 번 더 타자"는 동기 부여 효과가 발생했다. 업데이트 빈도 동적 조정으로 배터리 수명이 약 25% 연장되었으며, Dynamic Island 지원으로 앱이 최신 iOS 기능을 적극 활용한다는 긍정적 이미지를 형성했다. 사용자는 앱을 열지 않고도 필요한 정보를 즉시 확인할 수 있게 되어, 기술이 흐름을 방해하지 않고 자연스럽게 통합되었다.

## 04 목업 데이터 기반 개발 전략

외부 API 중단 상황에서 개발 연속성 확보

**문제**
2025년 9월 26일 국가정보자원관리원 화재로 기상청 공공데이터포털 API가 약 2주간 중단되었다. DoSurf는 실시간 파도, 바람, 기상 데이터에 전적으로 의존하는 서비스로, API 중단 시 신규 기능 개발과 테스트가 불가능해졌다. 차트 렌더링, 날씨 분류 로직, 데이터 저장 등 핵심 기능 개발이 모두 멈춰 2주 이상의 개발 일정 지연이 불가피한 상황이었다.

**접근**
API 중단 이전 수집한 실제 응답 구조를 분석하여 Firestore에 목업 데이터 세트를 구축했다. 프로토콜 기반 추상화 설계로 앱의 비즈니스 로직 계층이 데이터 출처를 인지하지 못하도록 격리했다. 데이터 접근 인터페이스를 프로토콜로 정의하고 실제 구현체가 이를 준수하도록 하여, ViewModel과 UseCase가 실제 API인지 목업 데이터인지 알 수 없게 만들었다. DTO는 기상청 API의 모든 필드를 Optional로 정의하여 실제 응답의 가변성을 반영했고, RxSwift 기반 비동기 처리 구조를 그대로 유지했다. 목업 데이터는 강릉, 포항, 제주, 부산 4개 지역 11개 해변의 3시간 간격 72시간 예보를 Firestore에 저장하여 실제 프로덕션 환경과 동일한 쿼리 방식을 보장했다.

**결과**
API 중단 기간 동안 개발을 멈추지 않고 차트 렌더링, 날씨 타입 분류, 서핑 기록 저장 등 핵심 기능을 완성했다. API 복구 후 데이터 수집 로직만 재개하면 되었고, 앱 코드는 단 한 줄의 수정 없이 실제 데이터로 전환되었다. 계층 간 의존성 주입을 통한 격리 설계가 데이터 소스를 교체할 때 기존 코드 수정 없이 전환 가능하게 만들었으며, 2주간의 잠재적 일정 지연을 완전히 회피하여 계획된 배포 일정을 준수했다. 외부 API 의존성을 추상화하는 설계의 중요성을 검증했고, 향후 예상치 못한 외부 시스템 장애 상황에서도 개발을 지속할 수 있는 대응 능력을 확보했다.
